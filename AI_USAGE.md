## 2. Where AI Was Used

We used AI tools in the following categories. In every case, the team member who used the AI reviewed, validated, and took responsibility for the output before committing it.

### 2.1 Code structure
- Generating consistent docstring templates across `src/` modules (parameter blocks, return types, notes sections). It should be noted that AI didn't write the docstrings, it helped format them.
- Repetitive try/except patterns around the data loaders (`src/yield_data_loader.py`, `src/data_loader_market.py`) (e.g., caching data).

### 2.2 Refactoring and code review
- Suggestions for extracting shared logic (e.g. the `_BinomialBase` parent class for the four binomial pricers, and the `_crr_setup` / `_crr_tree_price_*` decomposition in `src/derivative.py`).
- Identifying where the Portfolio class could delegate to standalone functions in `src/risk.py` rather than implementing risk metrics inline.
- Code review for readability — variable naming, consistent argument ordering, removing duplication.

### 2.3 Debugging
- Interpreting tracebacks for unfamiliar yfinance, pandas, and numpy errors.
- Diagnosing dimension-mismatch issues in the multi-asset Monte Carlo correlation logic (`05_trading_desk_analysis.ipynb`).
- Identifying that early-version finite-difference gamma on binomial trees was noise-dominated (resolved by using closed-form gamma from the Black-Scholes subclass for risk metrics).

### 2.4 Documentation prose
- Formatting summary and interpretation tables throughout the notebook. AI didn't write the information in these tables; it assisted in formatting them.

### 2.5 Visualisation
- matplotlib styling and construction suggestions (colour choices, grid formatting, spine removal) for consistency across notebooks.
- Built the more complex charts including multi-panel layouts, axis formatting, and the plotting code needed to render the scenario and risk visualisations.

## 3. Where AI Was NOT Used

The following work is entirely our own. AI tools were not used for the items below, and we can validate every choice independently.

### 3.1 Financial concepts and theory
- Understanding of Black-Scholes, the Merton dividend extension, the Cox-Ross-Rubinstein binomial tree, Monte Carlo pricing, Greeks, VaR, Expected Shortfall, Maximum Drawdown, and scenario analysis. These come from FINM3422 lecture materials, the supplementary reader, Investments (Bodie, Kane & Marcus), and Options, Futures, and Other Derivatives (Hull). Additionally, any other extensions that have been included were discussed briefly in our tutorials and were researched.
- All mathematical formulae used in the platform are transcribed from these course materials and textbooks, not generated by AI.

### 3.2 Portfolio design and rationale
- Equity selection. The choice of CBA, BHP, and CSL in `src/data_loader_market.py` — was made by the team based on sector diversification, ASX liquidity, and currency-consistency requirements for Black-Scholes validity against the RBA AUD curve. Justifications are documented in `docs/data_selection.md`.
- Strategy assignment per ticker. Pairing CBA with a protective put (hedging), BHP with a covered call (income), and CSL with a long ATM straddle (volatility speculation) was a deliberate design choice to demonstrate three distinct uses of options that were discussed in lectures.
- Position sizing. Share counts (600 / 1,700 / 1,000), contract counts (6 / 17 / 5), and hedge ratios (1:1 for CBA and BHP, partial for CSL) were chosen by the team to produce a portfolio with Greek profiles for meaningful risk analysis.
- Strike and maturity choices. 5% OTM strikes and 6-month maturities reflect typical market conventions; chosen by the team after reviewing Lecture 10 examples and weighing the trade-off between option premium and active hedge or income generation.

### 3.3 Model choices
- The decision to use Black-Scholes (with Merton extension) as the primary pricer for European options, and CRR as the cross-check, is grounded in the course's coverage.
- The decision to compute parametric VaR on dollar-delta-weighted exposure rather than raw portfolio value (to handle the option positions correctly) was a team decision documented in `05_trading_desk_analysis.ipynb`.
- The choice of historical VaR as the headline metric — with parametric VaR and MC VaR shown for comparison was based on the supplementary reader.

### 3.4 Numerical results
- Every number reported in the notebooks is generated by running our own code on the actual data. AI did not produce or estimate any quantitative result — it only helped us write the code that produced the numbers.
- Where AI suggested code snippets, we ran them, inspected the output, and confirmed it matched expectations before keeping the code.

### 3.5 Interpretation and discussion
- All written interpretation of results — what the Greeks mean for the portfolio, why the scenario P&Ls fall where they do, what the limitations are, what conclusions follow  was written by the team based our analysis.

### 3.6 Validation methodology
- Group members had to decide whether code was correct. For example, the binomial trees should converge to Black-Scholes for European options, that Monte Carlo should agree to within sampling error.

## 4. Validation of AI-Generated Code

Where AI tools produced code, we applied the following checks before accepting it into the codebase:

1. Manual read-through. No AI-generated block was committed without a team member reading every line and confirming they understood what it did.
2. Course-material reconciliation. Mathematical implementations were checked against the lecture slides and supplementary reader. Where the reader gave a formula (e.g. VaR_α = z_α · σ · V), we confirmed our code matched it line-by-line.
3. Numerical cross-validation. Pricing methods were validated against each other (Black-Scholes vs Binomial CRR vs Monte Carlo, all within tolerance). VaR methods were compared (historical vs parametric vs MC) to confirm they sit in defensible ranges. See `02_derivative.ipynb` and `05_trading_desk_analysis.ipynb`.
4. End-to-end execution. Every notebook was run top-to-bottom on at least one team member's machine before commit. Any execution failure was treated as a code defect, not an environment issue.
5. Independent reconstruction. For analytically important results (e.g. the closed-form put price, the protective-put payoff), at least one team member computed the value by hand on a small example to confirm the code matched.
